# Core Ledger — New Branch Implementation Brief

## Context

This document instructs an AI assistant to implement a new branch of the **Core Ledger** application from the `main` branch. The `main` branch is a Vite + React Router SPA generated by Lovable with shadcn/ui and Tailwind. The new branch must preserve the visual design and all functional screens, while adopting a clean architecture and migrating to Next.js with the **Pages Router**.

---

## Starting Point (main branch state)

The main branch has:
- **Vite** bundler, not Next.js
- **React Router v6** for client-side routing
- **Tailwind CSS v3** + **shadcn/ui** components (keep these — do not replace)
- **Open Sans** as the global font for all text
- Two monolithic data files: `src/lib/ca-data.ts` and `src/lib/paye-data.ts` (types + mock data + helpers all mixed together)
- A React Context (`src/context/fiscal-year.tsx`) for fiscal year state
- View components in `src/pages/` (these are not Next.js pages — they are React components used by React Router)
- No Zustand, no TanStack Query

---

## Target Stack

| Concern | Choice |
|---|---|
| Framework | Next.js 14 — **Pages Router** (not App Router) |
| Language | TypeScript |
| Styling | Tailwind CSS v3 + shadcn/ui (unchanged from main) + **SCSS Modules** for scoped component/view styles |
| State | **Zustand** for global state (fiscal year, current PAYE period) |

## What to Do

1. Create a new git branch from `main`.
2. Replace Vite with Next.js (Pages Router). Remove `vite.config.ts`, `src/main.tsx`, `src/App.tsx`, React Router dependencies.
3. Install: `next`, `zustand`, `@tanstack/react-query`, `sass`.
4. Set up the SCSS module infrastructure alongside Tailwind.
5. Refactor the monolithic data files into a clean architecture.
6. Implement all pages under `src/pages/` as proper Next.js Pages Router pages.
7. Re-introduce JetBrains Mono for figure classes while keeping Open Sans globally.

### Layer 3 — Repositories (`src/lib/repositories/`)

Data access only. Pure functions over mock data arrays. No formatting, no computation.

**`src/lib/repositories/ca.repository.ts`**

```ts
import { YEARS, CURRENT_FY } from "@/lib/mock-data/ca";

export const caRepository = {
  getAll: (): CAYear[] => YEARS,
  getByFiscalYear: (fy: number): CAYear | undefined => YEARS.find(y => y.fiscalYear === fy),
  getCurrentFiscalYear: (): number => CURRENT_FY,

```

**`src/lib/repositories/paye.repository.ts`**
```ts
import { EMPLOYEES, RUNS, CURRENT_PERIOD } from "@/lib/mock-data/paye";
import type { Employee, PayrollRun, Period } from "@/lib/models/paye";

export const payeRepository = {
  getEmployees: (): Employee[] => EMPLOYEES,
  getEmployeeById: (id: string): Employee | undefined => EMPLOYEES.find(e => e.id === id),
  getRuns: (): PayrollRun[] => RUNS,
  getRunByPeriod: (p: Period): PayrollRun | undefined =>
    RUNS.find(r => r.period.year === p.year && r.period.month === p.month),
  getCurrentPeriod: (): Period => CURRENT_PERIOD,
};
```

## Next.js Pages Router Setup

### Key differences from App Router (do not mix these up)

| App Router | Pages Router |
|---|---|
| `src/app/layout.tsx` | `src/pages/_app.tsx` |
| `src/app/page.tsx` | `src/pages/index.tsx` |
| `"use client"` directive | Not needed — all components are client-rendered by default |
| `useParams()` from `next/navigation` | `useRouter().query` from `next/router` |
| `useRouter()` from `next/navigation` | `useRouter()` from `next/router` |
| `<Link href="...">` | `<Link href="...">` (same, but from `next/link`) |
| Nested `layout.tsx` files | Per-page `getLayout` pattern or wrap in `_app.tsx` |

## Views to Implement

Migrate the existing views from `src/pages/` (React Router views) into the new Pages Router pages. The functional logic and visual design must be preserved exactly. Replace:
- `import { useNavigate, useParams, Link } from "react-router-dom"` → `import { useRouter } from "next/router"` and `import Link from "next/link"`
- `useFY()` context hook → `useFiscalYearStore()` Zustand hook
- Direct imports from `ca-data.ts` / `paye-data.ts` → hooks from `src/hooks/` and services from `src/lib/services/`
- Inline data access functions → service methods

Use SCSS modules for any custom scoped styles specific to a view or component. Use Tailwind for layout, spacing, colour utilities, and any styling that doesn't require SCSS features (nesting, variables, mixins).

**Rule of thumb:**
- If a style is a single Tailwind utility or two → use Tailwind class directly.
- If a style requires nesting, computed values, dynamic SCSS variables, or is more than 3 utilities on a single element → use an SCSS module class.
